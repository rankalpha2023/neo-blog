## 前言

过去一年里，我深度参与了多个 LLM 应用项目的开发和落地。这篇文章把**最值钱的教训**整理出来，希望能帮你少走弯路。

## 坑一：Prompt 不是越复杂越好

### 错误做法

很多人写 Prompt 的方式是"堆砌指令"：

```
你是一个超级智能的 AI 助手。你需要：
1. 仔细分析用户的问题
2. 从多个角度思考
3. 给出全面而深入的回答
4. 确保回答准确、清晰、有条理
5. 使用专业但易懂的语言
6. ...
（写了 500+ 行）
```

### 正确做法

**好的 Prompt 应该像给聪明同事写任务说明**：

```
你是一个技术文档撰写专家。
输入：功能需求描述
输出：结构化的技术文档（Markdown 格式）
要求：
- 每个章节独立成段
- 代码块标注语言类型
- 使用中文术语表
```

核心原则：**少即是多**。每多一条指令，模型就需要分配更多注意力去"理解规则"，反而降低了执行质量。

## 坑二：RAG 不是万能药

RAG（检索增强生成）确实能解决很多问题，但它不是银弹。

### RAG 适合的场景

- 知识库频繁更新（如产品文档、政策法规）
- 需要引用来源、可追溯
- 领域专业知识密度高

### RAG 不适合的场景

| 场景 | 为什么不适合 | 更好方案 |
|------|------------|---------|
| 通用问答 | 开源模型本身就知道 | 直接用模型 |
| 创意写作 | 检索会限制发散思维 | Few-shot + 温度调高 |
| 数学推理 | 向量检索不擅长逻辑 | Code Interpreter |
| 实时数据 | 向量库有延迟 | API + Function Calling |

## 坑三：忽略了评估体系

**没有评估 = 盲目优化。**

我见过太多团队花几周时间调优 RAG 系统，却从来没有建立过一套系统的评估方法。

### 最小可行评估框架

```python
# 三个维度就够了
eval_criteria = {
    "faithfulness": "回答是否基于检索到的内容？",
    "answer_relevance": "回答是否解决了用户的问题？",
    "context_relevance": "检索到的内容是否相关？"
}

# 每个维度 1-5 分，用 LLM-as-Judge 自动打分
```

> **关键洞察**：与其追求完美的评估指标，不如先建立一个"够用的"评估流程，然后持续迭代。

## 坑四：选型跟风

每个月都有新的框架、新的模型、新的方法出现。但你的项目需要的是**稳定可靠**的方案。

### 我的选型优先级

1. **先确定问题边界** — 到底要解决什么？
2. **从最简单的方案开始** — 能用 Prompt 解决就不用 RAG
3. **选择成熟方案** — 有社区、有文档、有案例
4. **预留切换成本** — 不要深度绑定某个特定模型

## 总结

LLM 应用开发的本质是 **系统工程**，不是魔法。把注意力放在数据质量、评估体系和用户体验上，比追逐最新模型更重要。
